指定範囲のテストケースひな形に、以下の処理フローからテストケースを入力してください。

# 生成ルール
- 処理、分岐、注記、共通ルールを読み取り、正常系・分岐系・異常系を網羅する。
- テストケースはフロー上の出現順で作成する。
- デフォルト状態では各処理は成功するものとする。
- 「<機能>に失敗」は、その機能に対応する関数実行の失敗として扱う。
- 共通の異常時処理がある場合は、各失敗パターンごとにテストケース化する。
- 分岐で通らない処理や異常時の後続処理は、期待結果に「実施されないこと」と明記する。
- フローチャート上に存在しない処理はテストケース化しない。ただし注記にある共通の異常時処理は対象とする。
- 明示されていない仕様や判断できない設定方法は、表の後に「確認事項」として列挙する。
- <br>は使わず、セル内改行で入力する。
- No 欄は数字のみ入力する。
- 空欄は作らない。該当なしは「-」、未実施結果は「NG」とする。

# 表記ルール
- No. は 1 から連番。
- 正常系の最初の手順は「1. 【デフォルト設定】を実施する」。
- 異常系の最初の手順は「1. 【○○失敗設定】を実施する」。
- 2手順目は「<title>を実行する」。
- フローチャート内の () は直前の機能に対応する関数。テスト観点・手順には関数名を含めず、期待結果には「機能(関数())」形式で記載する。
- 期待結果は「【動作ログ確認】」で始め、各項目は「・」始まり「こと」終わり。
- 異常系の期待結果は、実施済み機能 → 「失敗機能(関数())以降が実施されないこと」→「異常時処理が実施されること」→「戻り値 -1 で終了すること」の順。
