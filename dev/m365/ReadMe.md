3段階で十分だと思います。むしろ**人間が設計書を書く流れをそのまま再現**できるので、1回で全部やらせるより品質が上がります。

私なら以下の3段階にします。

---

# Step1：テンプレート理解（分析のみ）

**目的**

* テンプレートを理解する
* 編集は一切しない

**入力**

* 機能仕様書テンプレート（Word）
* 要件定義書（PDF）

**Copilotへの指示**

あなたはシステム設計書作成を支援するアシスタントです。

まず、開いている機能仕様書テンプレート全体を最後まで読み、テンプレートの構造を理解してください。

また、要件定義書も読み込み、記載内容を理解してください。

この段階では文書を一切編集してはいけません。

各章について以下を整理してください。

* 章の目的
* この章に記載すべき内容
* 要件定義書から転記・整理すればよい内容か
* 設計者の判断が必要な内容か
* 要件定義書だけでは不足しそうな内容

最後に一覧表としてまとめてください。

---

# Step2：本文生成

**目的**

* 本文だけ記入する

**入力**

* Step1の分析結果
* 機能仕様書テンプレート
* 要件定義書

**Copilotへの指示**

Step1の分析結果に従い、機能仕様書テンプレートへ本文を記入してください。

以下のルールを厳守してください。

【変更禁止】
・見出し
・章番号
・説明文
・記載要領
・スタイル
・レイアウト
・フォント
・表の構造
・セル結合
・ヘッダー
・フッター
・ページ設定

変更してよいのは本文のみです。

要件定義書に記載されている内容は忠実に整理・転記してください。

設計上の検討が必要な箇所は、一般的な設計内容を参考として記載し、「要確認」であることを明示してください。

要件定義書に記載がない事項は推測で断定せず、「要件定義書に記載なし」または「要確認」と記載してください。

文書全体の章構成やレイアウトは絶対に変更しないでください。

---

# Step3：レビュー

**目的**

* AI自身に成果物を点検させる

**入力**

* Step2で完成した仕様書

**Copilotへの指示**

完成した機能仕様書をレビューしてください。

以下の観点で確認してください。

1. テンプレートの見出し、章番号、レイアウト、スタイル、ヘッダー・フッターが変更されていないか。

2. テンプレートに元から記載されていた説明文や記載要領を削除・変更していないか。

3. 各章が記載要領を満たしているか。

4. 要件定義書に根拠のない内容を断定して記載していないか。

5. 「要確認」とすべき事項が漏れていないか。

6. 記載が不足している章はないか。

問題があれば該当箇所を修正してください。

修正後、どのような点を修正したかを箇条書きで報告してください。

---

## この構成の良いところ

この3段階は、人間の設計作業そのものです。

```text
要件定義書 + テンプレート
          │
          ▼
① テンプレート理解
（何を書く文書なのか理解）
          │
          ▼
② 本文作成
（本文だけを記入）
          │
          ▼
③ 自己レビュー
（品質チェック・不足点確認）
```

特にStep1が重要で、**いきなり書き始めるのではなく「まずテンプレートを読む」**ことを明示しているため、顧客ごとに異なるテンプレートでも適応しやすくなります。

### さらに改善するなら

もし運用してみて精度がもう一歩欲しい場合は、Step1の最後に次の一文を追加することをおすすめします。

> **Step2では、この分析結果を執筆方針として利用してください。分析結果と矛盾する記述は行わず、各章の目的を常に意識して本文を記述してください。**

これにより、Step1が単なる分析で終わらず、Step2の「設計方針」として機能するため、章ごとの品質がさらに安定しやすくなります。
